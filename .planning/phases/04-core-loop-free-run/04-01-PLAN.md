<feature>
  <name>RunEngineService + BPMTolerance</name>
  <files>BeatStep/Models/BPMTolerance.swift, BeatStep/Services/RunEngineService.swift, BeatStepTests/BPMToleranceTests.swift, BeatStepTests/RunEngineServiceTests.swift</files>
  <behavior>
    BPMTolerance:
    - BPMTolerance.tight.range == 3
    - BPMTolerance.normal.range == 7
    - BPMTolerance.loose.range == 12
    - BPMTolerance.defaultTolerance == .normal
    - save() persists to UserDefaults, BPMTolerance.saved reads it back
    - BPMTolerance conforms to CaseIterable for Picker usage

    RunEngineService - BPM Matching (findMatchingTracks):
    - 170 SPM with Normal tolerance matches songs at 163-177 BPM (direct)
    - 170 SPM also matches 78-92 BPM (half: 85 +/-7) and 333-347 BPM (double: 340 +/-7)
    - Songs without cached BPM are excluded from matches
    - Empty playlist returns empty matches

    RunEngineService - Fallback (findClosestTrack):
    - When no songs match within tolerance, returns the track with closest BPM to any target (SPM, SPM/2, SPM*2)
    - Never returns nil if playlist has any track with BPM data

    RunEngineService - No-Repeat Pool:
    - selectNextMatch marks selected track as played
    - Already-played tracks are excluded from future selections
    - When all matching tracks are played, pool resets and all become available again
    - Selection is random (not sequential)

    RunEngineService - Sustained Change Detection:
    - SPM change within tolerance range does NOT trigger re-match
    - SPM change outside tolerance range starts ~17s debounce timer
    - If SPM returns to old range before timer fires, timer cancels
    - If SPM stays at new level for full timer duration, commits as new sustained SPM and marks pending re-match
    - Re-match happens at next song end, not immediately (current song plays to completion)

    RunEngineService - Song-End Detection:
    - Monitors SpotifyPlayerService.currentTrack for ID changes
    - When track changes during active run, selects and plays next BPM-matched song
    - Race condition guard: concurrent song selections are serialized (isQueueingNext flag)

    RunEngineService - Run Lifecycle:
    - startRun(playlist:tracks:) loads BPMs into memory, plays first matched song, starts cadence monitoring and song-end monitoring
    - stopRun() cancels all monitoring tasks, resets played-track pool, clears state
    - During paused cadence state: keeps playing current song, does not pause music
  </behavior>
  <implementation>
    1. Create BPMTolerance.swift enum with Tight/Normal/Loose presets, range values, UserDefaults persistence via RawRepresentable String.

    2. Create RunEngineService.swift as @Observable singleton following project patterns:
       - @ObservationIgnored on all private stored properties (per project convention to avoid @Observable macro conflicts)
       - Load playlist BPMs into in-memory [String: Int] dictionary at run start (avoids repeated @MainActor BPMCacheService queries -- see Pitfall 4 in RESEARCH.md)
       - findMatchingTracks(forSPM:) checks SPM, SPM/2, SPM*2 against bpmMap within tolerance.range
       - findClosestTrack(forSPM:) returns track with minimum BPM distance to any target
       - selectNextMatch() filters out playedTrackIDs, picks random from matches, falls back to closest
       - Sustained change: Task.sleep(for: .seconds(17)) with cancellation on cadence revert
       - Song-end monitor: poll SpotifyPlayerService.currentTrack every 2s, detect ID change, queue next match
       - isQueueingNext Bool flag prevents race between skip and song-end (Pitfall 1)
       - Use play(uri:) WITHOUT contextURI so Spotify does not auto-advance (Pitfall 5)
       - Safety floor: never call play endpoint more than once per 5 seconds (rate limit protection)

    3. Make findMatchingTracks and findClosestTrack internal (not private) for direct unit testing (same pattern as CadenceService.processCadenceSample).

    4. Add new source files to project.yml targets (BeatStep sources and BeatStepTests sources).
  </implementation>
</feature>
