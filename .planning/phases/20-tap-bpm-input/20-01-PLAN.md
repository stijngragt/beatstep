<feature>
  <name>TapBPMEngine</name>
  <files>BeatStep/Services/TapBPMEngine.swift, BeatStepTests/TapBPMEngineTests.swift</files>
  <behavior>
    TapBPMEngine is an @Observable class with these published properties:
    - currentBPM: Int? (nil until second valid tap)
    - tapCount: Int (0-9, count of tap events including first)
    - isStable: Bool (true when intervals.count >= 8)
    - lastTapWasOutlier: Bool (true if most recent tap was rejected)
    - canSave: Bool (true when intervals.count >= 3, i.e. 4+ taps)

    Core behaviors to test (RED phase):
    1. First tap: tapCount=1, currentBPM=nil, canSave=false, isStable=false
    2. Second tap at 0.5s interval: currentBPM=120, tapCount=2
    3. Steady 120 BPM (0.5s intervals) x9 taps: currentBPM=120, isStable=true, tapCount=9
    4. Rolling window: after 12 steady taps, only last 8 intervals used (currentBPM still correct)
    5. Outlier rejection: after 4 steady 120 BPM taps, a 0.1s tap (600 BPM) is rejected -- lastTapWasOutlier=true, tapCount unchanged, currentBPM unchanged
    6. Boundary rejection: interval < 0.2s (too fast) or > 2.0s (too slow) rejected as outlier
    7. canSave: false at 0-3 taps, true at 4+ taps
    8. reset(): clears all state to initial values
    9. Inactivity reset: after 3s with no tap, state resets (tapCount=0, currentBPM=nil)
    10. Outlier with no prior intervals: second tap that's out of 0.2-2.0s range is rejected

    Implementation approach (GREEN phase):
    - Use Date() timestamps for tap timing (per research: sub-ms accuracy, no CADisplayLink needed)
    - Store tapTimestamps: [Date] and intervals: [TimeInterval]
    - Rolling window: keep last 8 intervals via removeFirst when count > 8
    - Outlier: compute median of current intervals, reject if abs(interval - median) / median > 0.40
    - Inactivity: Timer.scheduledTimer with 3s interval, [weak self], calls reset() on MainActor
    - tapCount tracks number of tap events (first tap = 1, each valid tap increments, outliers don't increment)
    - For testability: allow injecting a custom date provider or use tap(at: Date) overload

    NOTE: For unit testing the inactivity timer, provide a tap(at:) method that accepts a Date parameter. Tests can call tap(at:) with controlled timestamps. The 3s inactivity timer uses real Timer for production. Tests verify reset state directly after calling reset() -- timer behavior is implicitly tested by the production code path.
  </behavior>
  <implementation>
    Follow the research reference implementation closely (see 20-RESEARCH.md Pattern 1). Key deviations from research:
    - Add tap(at: Date) for testability alongside tap() that uses Date()
    - tapCount counts tap events (1-indexed), not intervals. First tap = 1, ninth valid tap = 9.
    - Progress dots in UI will map to intervals (0-8), but engine exposes tapCount for display as "N/8" where N = max(0, tapCount - 1) capped at 8.
  </implementation>
</feature>
