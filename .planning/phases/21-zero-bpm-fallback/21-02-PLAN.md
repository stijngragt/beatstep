    3. In `selectNextMatch(forSPM:)`, AFTER the existing `findClosestTrack` fallback returns nil (just before the final `return nil`), add the nil-BPM fallback logic:
       ```swift
       // Zero-BPM fallback: include nil-BPM tracks if user configured playRegardless or prompt
       if zeroBPMFallback == .playRegardless || zeroBPMFallback == .prompt {
           let nilBPMTracks = playlistTracks.filter { bpmMap[$0.id] == nil && !playedTrackIDs.contains($0.id) }
           if let track = nilBPMTracks.first {
               playedTrackIDs.insert(track.id)
               return track
           }
           // If nil-BPM pool exhausted, reset played set for nil-BPM tracks only and retry
           let allNilBPM = playlistTracks.filter { bpmMap[$0.id] == nil }
           if let track = allNilBPM.first {
               // Don't reset BPM-matched played IDs, only allow nil-BPM replay
               playedTrackIDs.insert(track.id)
               return track
           }
       }
       ```

    IMPORTANT: Do NOT modify `findMatchingTracks` or `findClosestTrack`. The fallback logic belongs ONLY in `selectNextMatch` after all BPM-based matching has been exhausted. This ensures BPM-matched tracks are always preferred.

    Do NOT make `selectNextMatch` async. The prompt case uses the same path as playRegardless for now (plays the track). A future phase can add the prompt UI overlay.
