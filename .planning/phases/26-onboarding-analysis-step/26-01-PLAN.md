**Layout (3 states):**

1. **Loading state** -- ProgressView while fetching playlists from Spotify API.

2. **Picker state** -- Shows after playlists load. Value-framed header (music.note.list icon in accent color, "Pick Your First Playlist" heading, "Choose a playlist to analyze. BeatStep will find the BPM for each track so it can match songs to your pace." body text). Below: a ScrollView of playlist rows (max ~8 visible, just name + track count, keep it simple -- no coverage badges since nothing is analyzed yet). Use the same cover art + text row pattern from PlaylistListView but simplified (no swipe actions, no coverage). Tapping a row starts analysis (state 3).

3. **Analyzing state** -- After user taps a playlist. Show the selected playlist name, a ProgressView with "Analyzing... X/Y tracks" text derived from `LibraryScanService.shared.scanProgress`. When `scanProgress` becomes nil (scan complete), show a checkmark with "Ready to run!" and enable the Continue button. The Continue button is a pill-shaped accent CTA matching other onboarding screens.

**Implementation details:**
- `@State private var playlists: [SpotifyPlaylist] = []`
- `@State private var isLoading = true`
- `@State private var selectedPlaylist: SpotifyPlaylist? = nil`
- `@State private var analysisComplete = false`
- Load playlists on `.task { }` using `SpotifyAPIService.shared.fetchPlaylists()` (first page only, 20 items is enough for picking)
- On playlist tap: set selectedPlaylist, call `await LibraryScanService.shared.scanPlaylistByID(playlist.id, name: playlist.name)`, then set analysisComplete = true
- Use design tokens consistently: Spacing, Radius, Color, Font extensions matching OnboardingSpotifyView/OnboardingHealthView
- Background: Color.surfaceBase
- Do NOT add a Skip button -- this step is required for the user to have a usable first run
