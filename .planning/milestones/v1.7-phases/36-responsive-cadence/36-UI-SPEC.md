> Visual and interaction contract for Phase 36: Responsive Cadence. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Phase Classification

**This is a behavioral-tuning phase, not a visual-change phase.** No new views, layouts, components, or visual elements are introduced. All changes are to service-layer constants and a new dead zone filter in `CadenceService.processCadenceSample()`. The existing `CadenceDisplayView` continues to render identically -- it simply receives updated `currentSPM` values with different timing characteristics.

The UI contract below documents what the existing display must **preserve** (no visual regressions) and the **behavioral expectations** the tuning must achieve.

## Behavioral Contract (Phase-Specific)

This section replaces the standard copywriting contract since Phase 36 introduces no new copy. Instead, it defines the behavioral expectations the tuning must achieve, as observed through the existing UI.

### Display Response Time

| Behavior | Before (Phase 35) | After (Phase 36) | Requirement |
|----------|--------------------|-------------------|-------------|
| SPM number updates after pace change | 5s rolling window | 2.5s rolling window | CAD-01: within 2 seconds |
| Song selection after sustained change | 17s debounce | 8s debounce | CAD-02: within 12 seconds total |
| SPM jitter during steady-state | no filter | 3 SPM dead zone | CAD-03: no jitter >5 SPM |

### Visual Preservation Rules

1. **No animation changes.** The cadence number, trend arrow, and sync label must continue using `BSAnimation.smooth` and `BSAnimation.quick` tokens. Do not add new transition animations for the dead zone filter.
2. **No layout changes.** `CadenceDisplayView` VStack/HStack structure and spacing tokens are unchanged.
3. **Dead zone is invisible.** The dead zone filter operates at the service layer (`CadenceService.currentSPM`). The view simply renders whatever `currentSPM` value it receives. There is no UI-side filtering or debouncing.
4. **Trend arrow independence.** The trend arrow (`trendArrow`) reads from `CadenceTrend` which is fed by `updateTrend(currentAvg:)` using the raw rolling average, NOT the dead-zone-filtered value. Trend arrows must remain responsive even when the SPM number is held stable by the dead zone.
